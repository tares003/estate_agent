| D-012 | **Cross-tenant FK reference on a user-supplied id** (surfaced in Phase B10). The buyer-enquiry form submits `propertyId` as a hidden field; `submitEnquiry` writes it to `Enquiry.propertyId` (a nullable FK to `Property.id`). RLS scopes the INSERT to the current tenant, but Postgres FK validation runs with RLS bypassed, so a crafted POST could link an enquiry (correctly in tenant A) to **another tenant's** property id. Low blast-radius (the enquiry itself is tenant-correct), but it is a tenant-isolation seam. | Medium | **RESOLVED** (Phase B12) | Raw migration `migrations/raw/0006_composite_tenant_fks.sql` re-points all 8 tenant-scoped child→parent relations onto **composite `(tenant_id, <fk>)` FKs** to `(tenant_id, id)` (with a UNIQUE `(tenant_id, id)` target on `branches`/`properties`), enforcing same-tenant references in the schema independent of RLS. Nullable relations use `ON DELETE SET NULL (<fk>)` (PG15+ column-list, preserves `tenant_id`); non-nullable CASCADE. Verified on pglite (PG16) in `src/composite-tenant-fks.test.ts` (24 cases incl. the single-column-FK vulnerability demo, all-8 cross-tenant rejection, UPDATE rejection, SET NULL/CASCADE). Migration adversarially reviewed across 4 lenses. Residual soft references → **D-016**. |
| D-013 | **Anti-spam (Cloudflare Turnstile) not yet wired on the public enquiry form** (Phase B10). CLAUDE.md §9 names Turnstile as the challenge-response layer "verified server-side on every form submission", and `@estate/ui` ships `AntiSpamChallenge`, but the EPIC-F enquiry form does not yet capture/verify a Turnstile token. No CI guard enforces Turnstile presence (G5 = consent, G8 = trust-marker), so this is a requirement gap, not a lint failure. | Medium | **RESOLVED** (Phase B13) | Added `apps/web/app/lib/turnstile.ts` — `verifyTurnstile()` over an injectable `TurnstileVerifier` (default posts to Cloudflare siteverify; fails closed; **operator-level** keys via `TURNSTILE_SECRET_KEY` / `NEXT_PUBLIC_TURNSTILE_SITE_KEY`, documented for a future per-tenant swap). `submitEnquiry` verifies the token (with request IP) **before** the `withTenant` write — failure persists nothing. `EnquiryForm` renders `AntiSpamChallenge` + the `cf-turnstile-response` field when a sitekey is configured. Reusable by the viewing/valuation/repair forms. Cloudflare/Turnstile already declared in `docs/sub-processors.json` (G10). |
| D-014 | **No multiline Textarea atom in `@estate/ui`** (Phase B10). The enquiry "Message" field uses the single-line `TextField` because the EPIC-L component set has no Textarea primitive. Functional and accessible, but a long message is cramped. | Low | **OPEN** | Add a `Textarea` atom to `packages/ui` (token-driven, label/hint/error wired like `TextField`) and switch the message field to it. No build impact. |
| D-018 | **`@updatedAt` columns have no database default** (surfaced by the B16 real-Postgres integration run). Every model's `updatedAt` maps to a `NOT NULL` column with no DB default — Prisma sets the value in the client, so app/ORM writes are fine, but a **raw SQL `INSERT`** (seed scripts, or a raw tenant-provisioning `INSERT` per the §S.2 "INSERT a tenant row" claim) must supply `updated_at` or it fails the NOT-NULL constraint. | Low | **OPEN** | App code uses Prisma (unaffected). If any provisioning/seed path uses raw SQL, supply `updated_at = now()` (as the integration seed does), or add `@default(now())` to the `updatedAt` fields via a schema amendment so raw inserts work unchanged. No build impact. |
| D-017 | **Catalogue listing-type filter is not yet pack-gated** (Phase B14). The design brief (§C / design-requirements §2a) says the listing-type filter should offer only the tenant's enabled packs; the V1 filter bar offers all six listing types. This is **not a capability leak** — pack-gated properties are not in public lists regardless, so the filter merely narrows already-visible inventory (a disabled-pack type simply matches nothing). It is a UX refinement. | Low | **OPEN** | Wire `isPackEnabled` into `PropertyFilters` (and the listing-type→pack-slug map) once EPIC-AD entitlement reads from the session, to hide options for packs the tenant has not enabled. The search-param parser stays core (`// pack: core`) and keeps recognising every value. |
| D-016 | **Soft Agent references are not tenant-FK-enforced** (surfaced by the D-012 adversarial review, Phase B12). `notes.author_agent_id` and `property_status_events.changed_by_agent_id` are optional links to `Agent` with **no** Prisma `@relation` (by design — a note / status event must survive the author agent leaving), as is `agents.user_id`. They are set **server-side from the acting session, never from untrusted client input**, so the D-012 user-supplied-id vector does not currently apply — and adding a hard composite FK would contradict the deliberate "soft reference" design. | Low | **OPEN** | Documented in `0006_composite_tenant_fks.sql`. If any of these ever becomes **user-settable**, validate its tenancy in the application layer (resolve the agent within `withTenant` and reject a non-match) or add a matching composite `(tenant_id, <fk>)` FK. No action needed while server-set. |
| D-015 | **Viewing-request shape diverges three ways** (surfaced approaching Phase B11). Master spec row 25 "Book a viewing" specifies fields **Name, email, phone, date, time-slot** and behaviour "**Lead + viewing request created**; email to assigned agent + branch; confirmation + ICS to applicant". But (a) `packages/validators/.../viewing-request.ts` instead has `preferredDate` + `alternativeDate?` + `message?` (no time-slot); (b) the `Viewing` Prisma model has a single `preferredAt` DateTime + `notes?` (no alternative-date column, no time-slot band); (c) the dual lead+viewing write and the agent/branch email + applicant ICS are worker/email-template concerns (EPIC-U + EPIC-D), not yet built. Building the viewing action now would require inventing the field→column mapping. | Medium | **OPEN** | Owner/spec decision before B11 viewing flow. Recommended default (spec is the source of truth, CLAUDE.md §1): align the validator to **date + time-slot** (using the existing `TimeSlotSelector` atom + configurable bands), combine into `preferredAt`, and **defer** the dual-enquiry-write + email/ICS notifications to the EPIC-U/EPIC-D waves with the viewing row carrying enough to drive them later. Not yet ratified — pending direction. |
